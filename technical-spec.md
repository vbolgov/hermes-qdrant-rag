# Техническое задание: локальная база знаний Hermes + Qdrant RAG

**Статус:** фактическое ТЗ текущей реализации  
**Версия:** 1.1  
**Проект:** `hermes-qdrant-rag`

---

## 1. Цель

Создать локальную, воспроизводимую базу знаний для Hermes.

Система должна:

1. хранить исходные документы локально;
2. извлекать из них текст, делить его на фрагменты и создавать embeddings;
3. хранить векторы и метаданные в локальном Qdrant;
4. давать Hermes доступ к поиску через MCP;
5. возвращать найденные фрагменты вместе с источником;
6. поддерживать локальные и внешние embeddings;
7. обновлять индекс инкрементально: добавлять и переиндексировать только изменённые документы, удалять векторы удалённых файлов.
8. вести документ с уже индексированными RAG-документами и определять изменения по SHA-256 содержимого, чтобы не индексировать заново неизменённые документы.

---

## 2. Пользовательский сценарий

1. Пользователь кладёт документы в `library/`.
2. Пользователь запускает индексатор через Hermes skill `rag_update`.
3. Индексатор извлекает текст, нарезает его на фрагменты, создаёт embeddings и сохраняет их в Qdrant.
4. Hermes сообщает, что RAG готов к работе, если была выполнена переиндексация.
5. Пользователь просит Hermes найти сведения в локальной базе через skill `rag_search`.
6. `rag_search` проверяет актуальность индекса. Если индекс актуален, skill выполняет поиск. Если требуется индексация, Hermes показывает список новых, изменённых и удалённых документов и спрашивает пользователя, запускать ли skill `rag_update`.
7. Hermes вызывает MCP-инструмент `qdrant-rag`.
8. MCP возвращает релевантные фрагменты с путём к исходному файлу, названием документа, номером фрагмента и score.
9. Hermes использует найденные фрагменты при ответе с цитатами из RAG.

## 3. Границы проекта

### Входит в первую версию

- Локальный Qdrant в Docker / OrbStack.
- MCP-интеграция с Hermes.
- Индексация `.md`, `.txt`, `.pdf`, `.docx`, `.epub`, `.fb2`.
- Embeddings через:
  - локальную модель FastEmbed;
  - OpenAI API;
  - детерминированный hash fallback для тестов.
- Инкрементальная синхронизация и manifest индекса.
- Метаданные из YAML frontmatter Markdown-файлов.
- Исключение файлов через `.ragignore`.
- Проверка состояния коллекции и поиск через MCP.
- Локальное OCR сканированных PDF без текстового слоя после подтверждения пользователя.

### Не входит в первую версию

- Автоматический импорт из облачных хранилищ, Notion, Google Drive и почты.
- Автоматическая переиндексация по watcher/cron.
- Граф знаний.
- Веб-интерфейс управления документами.
- Аутентификация Qdrant для локального сценария.
- Мультипользовательский доступ и разграничение прав.

## 4. Архитектура

```text
Исходные документы
       │
       ▼
library/ ──► ingest.py ──► embeddings.py ──► Qdrant
                                              │
                                              ▼
                                      rag_mcp.py (MCP)
                                              │
                                              ▼
                                           Hermes
```

### Компоненты

| Компонент | Ответственность |
|---|---|
| `library/` | Архив исходных документов. Источники не изменяются системой. |
| `ingest.py` | Извлечение текста, chunking, создание embeddings, синхронизация Qdrant. |
| `embeddings.py` | Выбор и реализация embedding backend. |
| `index_state.py` | Хеширование файлов, manifest, определение добавленных/изменённых/удалённых документов. |
| `rag_mcp.py` | MCP-сервер для Hermes. |
| `docker-compose.yml` | Локальный Qdrant. |
| `Context/rag-index-manifest.csv` | Состояние последней индексации. |
| `.env` | Секреты и локальная конфигурация. Не коммитится. |

## 5. Хранилище и окружение

### 5.1 Qdrant

- Запускается локально через Docker / OrbStack.
- Имя контейнера: `hermes-qdrant`.
- HTTP endpoint: `http://localhost:6333`.
- Данные должны сохраняться в Docker volume и переживать перезапуск контейнера.
- Политика перезапуска: `unless-stopped`.

### 5.2 Коллекция

По умолчанию:

```env
QDRANT_COLLECTION=consulting_kb
```

Коллекция создаётся автоматически при первой индексации.

Метрика близости: cosine distance.

### 5.3 Документы

Корень документов по умолчанию:

```text
library/
```

Разрешены вложенные папки. Путь к файлу внутри `library/` сохраняется как `source_path`.

Поддерживаемые форматы:

| Формат | Способ извлечения |
|---|---|
| `.md`, `.txt` | Чтение UTF-8 текста |
| `.pdf` | PyMuPDF для PDF с текстовым слоем; локальное OCR для сканов после подтверждения пользователя |
| `.docx` | python-docx |
| `.epub` | EbookLib + BeautifulSoup |
| `.fb2` | XML parser |

Сканированные PDF без текстового слоя должны быть определены до индексации. Hermes показывает пользователю список таких файлов, предупреждает, что OCR может занять время, и запускает локальное OCR только после явного подтверждения пользователя.

## 6. Embeddings

Поддерживаются три backend:

| Backend | Переменная | Назначение |
|---|---|---|
| Локальный | `EMBEDDING_BACKEND=local` | Основной режим без внешнего API. Использует FastEmbed. |
| OpenAI | `EMBEDDING_BACKEND=openai` | Внешний embedding через OpenAI API. |
| Hash | `EMBEDDING_BACKEND=hash` | Детерминированный fallback для тестов; не предназначен для качественного семантического поиска. |

### 6.1 Настройка OpenAI embeddings

В `.env` должны быть:

```env
EMBEDDING_BACKEND=openai
OPENAI_API_KEY=...
OPENAI_BASE_URL=https://api.openai.com/v1
OPENAI_EMBED_MODEL=text-embedding-3-small
```

`OPENAI_BASE_URL` позволяет использовать OpenAI-compatible embedding API.

Если backend, embedding-модель или размерность векторов изменены после существующей индексации, Hermes должен сообщить пользователю старую и новую конфигурацию, предложить полную переиндексацию и запускать `--rebuild` только после явного подтверждения пользователя.

### 6.2 Настройка локальных embeddings

```env
EMBEDDING_BACKEND=local
LOCAL_EMBED_MODEL=sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2
```

Локальная модель должна быть доступна FastEmbed. При первом использовании допускается загрузка модели.

## 7. Индексация

### 7.1 Chunking

- Максимальная длина фрагмента: `1800` символов.
- Перекрытие соседних фрагментов: `250` символов.
- Предпочтительные границы: абзац, затем перенос строки.
- Пустые фрагменты не сохраняются.

### 7.2 Инкрементальная синхронизация

Для каждого документа система хранит:

- относительный `source_path`;
- SHA-256 файла;
- время изменения;
- размер;
- дату индексации;
- число созданных точек.

При обычном запуске:

- новые и изменённые файлы переиндексируются;
- неизменённые файлы пропускаются;
- удалённые файлы удаляются из Qdrant;
- manifest обновляется атомарно.

### 7.3 Управление индексом

Поддерживаются команды:

```bash
# Показать план синхронизации без записи в Qdrant
python ingest.py --dry-run

# Обычная инкрементальная индексация
python ingest.py

# Полностью пересоздать коллекцию и индекс
python ingest.py --rebuild
```

### 7.4 Skills управления RAG

#### `rag_search`

1. Запускает `ingest.py --dry-run` и проверяет актуальность индекса по manifest и SHA-256.
2. Если индекс актуален, выполняет поиск через MCP.
3. Если индекс устарел, показывает список новых, изменённых и удалённых документов и спрашивает пользователя, запускать ли `rag_update`.
4. Не запускает переиндексацию самостоятельно без подтверждения пользователя.

#### `rag_update`

1. Показывает план изменений перед записью в Qdrant.
2. Запрашивает подтверждение перед удалением векторов и перед `--rebuild`.
3. Для сканированных PDF показывает список файлов и запускает OCR только после подтверждения пользователя.
4. Запускает индексацию.
5. Проверяет `collection_stats`.
6. Сообщает число добавленных, обновлённых и удалённых документов и созданных фрагментов.

## 8. Метаданные и источники

### 8.1 Обязательные payload-поля Qdrant

Каждый фрагмент должен содержать:

```text
source_id
source_path
title
chunk_id
chunk_index
source_sha256
embedding_mode
embedding_model
text
```

### 8.2 Опциональные метаданные Markdown

Markdown-файлы могут содержать YAML frontmatter:

```yaml
---
source_id: unique-source-id
title: Название документа
source_type: article
domain: consulting
reliability: primary
date_added: 2026-07-10
tags: [rag, knowledge]
marker: important
---
```

Разрешённые дополнительные поля, сохраняемые в payload:

```text
source_type
domain
reliability
date_added
tags
marker
```

### 8.3 Исключение файлов

Файл `library/.ragignore` должен работать аналогично упрощённому `.gitignore`: каждая строка — pattern относительного пути, который не попадает в индекс.

## 9. MCP-интеграция Hermes

MCP-сервер: `rag_mcp.py`.

Имя сервера:

```text
qdrant-rag
```

### 9.1 Инструменты

#### `collection_stats`

Должен вернуть:

```text
qdrant_url
collection
exists
point_count
embedding_mode
embedding_model
data_dir
```

#### `search_documents(query, top_k=5)`

Должен:

1. создать embedding поискового запроса тем же backend, которым создан индекс;
2. выполнить векторный поиск в Qdrant;
3. ограничить `top_k` диапазоном от 1 до 50;
4. вернуть список фрагментов с score и источником.

Формат результата каждого фрагмента:

```text
score
source_id
source_path
title
chunk_id
chunk_index
text
```

## 10. Правила использования знаний

### RAG

RAG используется для:

- конкретного факта;
- цитаты;
- поиска упоминания;
- работы с большим архивом;
- первичных гипотез.

### Wiki

Wiki используется для:

- глубокого анализа;
- истории решений;
- причинности;
- экспертной корректировки.

### Гибрид

RAG и Wiki используются совместно, если:

- требуется высокая точность;
- источников несколько;
- есть конфликт сведений;
- есть сомнения, какой слой использовать.

Все существенные выводы должны сопровождаться:

- цитатой или фрагментом источника;
- путём к источнику;
- качественным уровнем уверенности.

Правила уровня уверенности:

- **высокая** — прямой факт есть в источнике;
- **средняя** — вывод собран из нескольких фрагментов; Hermes показывает краткие цитаты;
- **низкая** — это гипотеза или данных недостаточно; Hermes показывает, на каких кратких цитатах основана гипотеза.

## 11. Безопасность и переносимость

1. Исходные документы не изменяются индексатором.
2. Секреты хранятся только в `.env`.
3. `.env`, данные Qdrant, кэши и логи не коммитятся.
4. В коде и документации не должны появляться абсолютные пользовательские пути; используются относительные пути от корня проекта или переменные окружения.
5. API-ключи не выводятся в логи, ответы MCP и manifest.

- [ ] OrbStack / Docker доступен.

## 13. Риски и ограничения

| Риск / ограничение | Последствие | Действие |
|---|---|---|
| Нет OpenAI API-ключа | Невозможен backend `openai` | Использовать `local` или добавить ключ в `.env`. |
| Смена embedding backend | Несовместимая размерность векторов | Показать пользователю старую и новую конфигурацию; запускать `ingest.py --rebuild` только после подтверждения. |
| PDF является сканом | Нет извлечённого текста без OCR | Показать список сканов, предупредить о времени обработки и запускать OCR только после подтверждения пользователя. |
| Слишком большой архив | Долгая первая индексация | Использовать инкрементальный режим после первой загрузки. |
| Низкая точность hash backend | Слабый семантический поиск | Использовать только для тестов. |
| Qdrant выключен | MCP-поиск недоступен | Запустить контейнер через Docker / OrbStack. |

## 14. Результат

После реализации пользователь получает локальный слой знаний для Hermes:

- документы остаются на его Mac;
- Qdrant хранит восстановимый векторный индекс;
- embeddings могут быть локальными или внешними;
- Hermes обращается к базе через MCP;
- ответы можно строить на конкретных фрагментах с указанием источника.
